Before making any code changes, provide a clear explanation covering the following three areas for the file or component we are about to modify. The explanation must be thorough yet accessible:

## 1. What Are We About to Modify?

Identify the specific file(s), component(s), or section(s) that will be changed. Describe:
- The current state of the code (what it does right now)
- What exactly we plan to change and why
- Which parts of the codebase will be affected (direct and indirect impacts)

## 2. Why Is This Important?

Explain the significance of this modification:
- What problem does it solve or what value does it add?
- What would happen if we did NOT make this change?
- How does it fit into the larger goals of the project?
- Are there any risks or trade-offs to be aware of?

## 3. Analogy for Better Understanding

Provide a relatable, real-world analogy that makes the concept easy to grasp — even for someone who is not deeply technical. The analogy should:
- Map clearly to what we are doing in the code
- Highlight why the change matters
- Be memorable and intuitive

---

**Format your response with clear headings and keep the language conversational.** Do NOT start coding until this explanation is delivered and acknowledged by the user. Ask the user: *"Does this make sense? Should I proceed with the changes?"*

Use `$ARGUMENTS` as context for what the user wants to modify. If no arguments are provided, ask the user what they would like to work on.
